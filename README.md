# 数据中心智能运维支撑平台

## 项目定位

数据中心智能运维支撑平台面向机房、服务器、网络设备和自动化脚本等运维场景，提供统一的账号认证、权限控制、任务执行入口、操作留痕和后续审计能力。

当前代码库处于最小可运行骨架阶段，已经打通用户登录、JWT 令牌签发、当前用户查询、基于 RBAC 的权限校验、统一响应、统一异常处理和 traceId 追踪链路。后续功能会围绕真实运维流程继续扩展，例如资源台账、脚本管理、任务执行、执行日志、审批流和风险控制。

## 已实现能力

- 用户登录：支持账号密码登录，密码使用 BCrypt 哈希校验。
- JWT 认证：登录成功后签发访问令牌，后续接口通过 `Authorization: Bearer <token>` 传递身份。
- 当前用户：提供接口返回当前登录用户、角色编码和权限编码。
- RBAC 授权：通过用户、角色、权限三类数据控制接口访问范围。
- 401 / 403：未登录统一返回 401，已登录但权限不足统一返回 403。
- 统一响应：所有业务接口使用统一 JSON 结构返回 `code`、`message`、`data` 和 `traceId`。
- 统一异常：参数错误、业务错误、认证失败、授权失败和系统异常集中处理。
- traceId：每个请求生成追踪编号，便于接口排查和日志串联。
- 本地数据库：使用 H2 内存库初始化用户、角色和权限数据，方便快速启动。
- 自动化测试：使用 JUnit 5 和 MockMvc 验证登录、鉴权、401、403 等核心链路。

## 技术栈

| 类型 | 技术 | 当前用途 |

| 测试框架 | JUnit 5、MockMvc | 接口级自动化回归验证 |

```bash

```

```bash

```

服务默认启动在：

```text
http://localhost:8080
```

```bash
mvn test
```

## 本地数据库

项目默认使用 H2 内存数据库，应用启动时会自动执行 `schema.sql` 和 `data.sql`。

H2 控制台地址：

```text
http://localhost:8080/h2-console
```

连接信息：

| 配置项 | 值 |
| --- | --- |
| JDBC URL | `jdbc:h2:mem:ops_platform` |
| User Name | `sa` |
| Password | 空 |

内存库的数据会随进程结束而释放，适合本地快速启动和自动化测试。生产环境需要切换为外部数据库，并通过环境变量、配置中心或密钥管理系统注入连接信息。

## 接口说明

### 登录

```http
POST /api/v1/auth/login
Content-Type: application/json
```

请求示例：

```json
{
  "username": "admin",
  "password": "123456"
}
```

响应中的 `token` 用于访问后续受保护接口。

### 当前用户

```http
GET /api/v1/auth/me
Authorization: Bearer <token>
```

用于验证当前令牌对应的用户身份、角色和权限。

### 模拟执行任务

```http
POST /api/v1/tasks/{id}/mock-execute
Authorization: Bearer <token>
```

该接口需要 `task:execute` 权限。`admin` 可以访问，`viewer` 会返回 403。

## 响应格式

成功响应示例：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "username": "admin"
  },
  "traceId": "b8d9f7c6a1e24a3a"
}
```

失败响应示例：

```json
{
  "code": 403,
  "message": "无权访问当前资源",
  "data": null,
  "traceId": "b8d9f7c6a1e24a3a"
}
```

`traceId` 会贯穿单次请求，后续接入日志系统后可用于快速定位问题。

## 关于 Redis 和消息队列

当前版本没有引入 Redis、消息队列或本地缓存组件，原因是现有能力还没有出现必须依赖这些组件才能解决的问题。平台会按照业务压力和故障边界逐步引入基础设施，而不是提前堆叠组件。

后续出现以下场景时再引入对应技术：

| 场景 | 候选技术 | 引入原因 |
| --- | --- | --- |
| 高频读取任务状态、用户权限、资源快照 | Redis / Caffeine | 降低数据库压力，提升热点读取速度 |
| 任务提交后需要快速返回，执行过程较长 | 消息队列 | 解耦接口请求和后台执行，提高吞吐和可靠性 |
| 脚本任务需要失败重试、延迟执行、异步通知 | 消息队列 | 统一处理重试、削峰和事件流转 |
| JWT 需要主动失效、账号被禁用后立即踢出 | Redis | 维护 token 黑名单或登录态版本号 |
| 多实例并发领取同一个任务 | Redis 分布式锁 / 数据库乐观锁 | 避免任务被重复执行 |

## 开发约定

- 所有 Java、SQL、YAML、脚本和项目文档都使用中文说明关键设计意图。
- 配置文件不能写入真实服务器地址、真实账号、真实密码、真实令牌或其他敏感信息。
- 接口新增或权限规则变化时，需要同步补充自动化测试。
- 新增第三方组件前，需要先说明它解决的具体业务问题、故障边界和替代方案。
- 目录结构优先保持清晰可维护，只有在复杂度上升时才拆分更重的工程结构。
